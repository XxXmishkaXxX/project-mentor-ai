# ВВЕДЕНИЕ
В рамках проекта рассматривается разработка ИИ-ассистента для студентов, который помогает вести учебные и проектные работы. Основная идея системы состоит в том, что ассистент не должен отвечать только на основе общих знаний языковой модели. Он должен находить подходящие фрагменты в заранее подготовленной базе знаний и на их основе формировать ответ для пользователя.

Такой подход важен для учебной среды, потому что студентам часто нужна не просто справка, а понятная подсказка по методологии: как составить техническое задание, как распределить роли, как описать требования, как вести backlog, как подготовить документацию и как оценивать результат проекта. Поэтому в качестве основной архитектуры выбрана RAG-архитектура, то есть связка поиска по базе знаний и генерации ответа большой языковой моделью.

Цель аналитики - определить, какие технологические подходы, инструменты и функции целесообразно включить в первую итерацию проекта, а какие решения лучше отложить. Структура анализа сохранена по примеру: сначала рассматриваются технологические варианты, далее приводятся сравнительные таблицы, затем анализируются аналоги и формируются выводы для первой итерации.

# 1 АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ
## 1.1 Общая характеристика проекта
Проект относится к прикладным LLM-приложениям. Его результатом должен стать веб-сервис или чат-бот, который консультирует студентов по вопросам проектной деятельности. В качестве источников знаний предполагается использовать статьи, книги, методические материалы, стандарты, материалы по Agile, Scrum, DevOps, управлению требованиями и ведению документации.

Ключевое отличие проекта от обычного чат-бота состоит в контроле источников. Обычная языковая модель может давать уверенные, но неточные ответы. В RAG-системе ответ строится после поиска релевантных фрагментов в базе знаний. Официальная документация LangChain описывает RAG как усиление ответов LLM за счет контекстной информации, полученной во время запроса [1].

## 1.2 Основная проблема пользователей
Студенты часто сталкиваются с организационными и методологическими вопросами. Например, они могут понимать, какую программу нужно написать, но не всегда понимают, как оформить цель проекта, как разбить работу на задачи, как вести документацию, как проводить итерации и как показывать результат заказчику.

Из-за этого в проекте важно сделать ассистента не просто справочником, а помощником, который дает короткий и понятный ответ, предлагает структуру документа, показывает пример и ссылается на источник. Для первой итерации достаточно реализовать прямые ответы по базе знаний, но уже на этом этапе нужно заложить проверку качества ответов.

## 1.3 Обоснование RAG-архитектуры
RAG подходит для данного проекта, потому что база знаний может регулярно дополняться без полного переобучения модели. Документы разбиваются на фрагменты, для них создаются векторные представления, после чего система находит наиболее близкие фрагменты к вопросу пользователя. LlamaIndex в своей документации описывает такой процесс как этап индексации, при котором данные преобразуются в embeddings и сохраняются в vector store [2].

Для учебного ассистента это дает три преимущества: ответы можно ограничивать проверенными источниками, новые материалы можно добавлять постепенно, а качество можно оценивать на наборе тестовых вопросов. Поэтому RAG является оптимальным базовым подходом для первой версии системы.

# 2 АНАЛИЗ ТЕХНОЛОГИЧЕСКИХ ПОДХОДОВ
## Классический RAG
**Архитектура:** Система состоит из базы документов, этапа разбиения на фрагменты, модели embeddings, векторной базы данных, ретривера и LLM. Пользователь задает вопрос, ретривер ищет подходящие фрагменты, после чего LLM формирует ответ с учетом найденного контекста.

**Качество ответов:** Хорошо подходит для прямых вопросов по базе знаний: например, «Как составить ТЗ?» или «Что такое sprint backlog?». Качество зависит от полноты базы, размера фрагментов, качества embeddings и корректности промпта.

**Скорость:** Обычно работает быстро, так как на один запрос выполняется поиск по векторной базе и один вызов LLM. Для учебного прототипа это самый простой и понятный вариант.

**Контроль и кастомизация:** Можно управлять источниками, шаблоном ответа, количеством найденных фрагментов, порогом релевантности, языком ответа и форматом ссылок.

**Применение в проекте:** Основной вариант для минимального и базового результата проекта. Позволяет показать цепочку «запрос - поиск - ответ» и быстро перейти к работающему прототипу.

**Ограничения:** Плохо справляется с вопросами, где нужно объединять знания из многих документов, если не настроены reranking, metadata filtering и контроль источников.

## Agentic RAG
**Архитектура:** Кроме обычного поиска, система использует агента, который может сам выбирать действие: уточнить вопрос, выполнить поиск, обратиться к инструменту, проверить ответ или переформулировать запрос.

**Качество ответов:** Может давать более точные ответы на сложные вопросы, потому что не ограничивается одним поиском. Например, агент может отдельно найти материал про роли Scrum и отдельно материал про оформление задач.

**Скорость:** Медленнее классического RAG, потому что может выполнять несколько шагов и несколько обращений к модели.

**Контроль и кастомизация:** Гибкость выше: можно добавить инструменты проверки, генерации шаблонов, классификации вопроса и сбора обратной связи.

**Применение в проекте:** Подходит для оптимального результата, когда уже есть базовый ассистент и нужно сделать его более похожим на ментора.

**Ограничения:** Для первой итерации может быть избыточным, так как сложнее тестировать, логировать и объяснять работу системы.

## GraphRAG
**Архитектура:** Система строит граф знаний по документам: выделяет сущности, связи и тематические сообщества. Microsoft описывает GraphRAG как подход, объединяющий извлечение текста, сетевой анализ, LLM-промптинг и суммаризацию [3].

**Качество ответов:** Хорошо подходит для сложных вопросов, где нужно видеть связи между понятиями. Например, связь между требованиями, ролями, артефактами Scrum и критериями приемки.

**Скорость:** Обычно медленнее и дороже на этапе подготовки базы, потому что нужно извлекать сущности и связи. Ответы также могут требовать более сложной логики поиска.

**Контроль и кастомизация:** Позволяет управлять связями между понятиями и строить более объяснимые ответы. Можно показывать, какие темы и документы связаны с вопросом.

**Применение в проекте:** Перспективен для расширенной версии ассистента, если база знаний станет большой и появится необходимость отвечать на комплексные вопросы.

**Ограничения:** Сложнее в разработке и поддержке. Для минимального результата проекта не обязателен.

## Fine-tuning языковой модели
**Архитектура:** Модель дополнительно обучается на примерах вопросов и ответов или на доменных данных. В отличие от RAG, знания частично переносятся внутрь модели.

**Качество ответов:** Может улучшить стиль ответов и следование формату, но не гарантирует точность фактов и не заменяет проверяемую базу знаний.

**Скорость:** После обучения ответы могут быть быстрыми, но само обучение требует подготовки датасета, вычислительных ресурсов и проверки качества.

**Контроль и кастомизация:** Хорошо настраивает тон, формат и типовые сценарии, но хуже подходит для постоянно обновляемой базы учебных материалов.

**Применение в проекте:** Может быть использован позже для улучшения стиля ответов, когда накопятся реальные диалоги и оценки пользователей.

**Ограничения:** Не является лучшим вариантом для первой итерации, так как проекту требуется ответ на основе проверенных источников, а не просто имитация стиля.

## Long-context LLM без отдельной базы
**Архитектура:** В запрос модели помещается большой объем текста: методичка, стандарт или набор документов. Отдельный поиск может не использоваться.

**Качество ответов:** Подходит для разового анализа небольшого набора документов. Если контекст очень большой, модель может пропускать важные части или давать ответ без точной привязки к источнику.

**Скорость:** Скорость зависит от размера контекста. Чем больше документов отправляется в модель, тем выше задержка и стоимость.

**Контроль и кастомизация:** Настройка простая, но контроль слабее, чем у RAG: сложнее управлять обновлением базы, версиями документов и источниками.

**Применение в проекте:** Может использоваться для прототипирования или проверки отдельных материалов.

**Ограничения:** Не подходит как основная архитектура проекта, потому что база знаний должна быть структурированной, обновляемой и проверяемой.

## 4.3 Оценка качества ответов
В ТЗ указано, что качество ассистента должно быть оценено на наборе из 20+ тестовых вопросов. Для этого нужно заранее подготовить вопросы, эталонные ответы или критерии оценки, а затем проверять релевантность, полноту, корректность и наличие опоры на источник. Ragas описывает подход к систематической оценке LLM-приложений, а LangSmith отдельно показывает сценарии оценки RAG-приложений на тестовых наборах [9], [10].

| Notion AI                   | Зависит от тарифа Notion                          | Высокая             | Хорошее внутри рабочей области       | Работает с контентом workspace и enterprise search [13]        | Удобен для командной документации и заметок                  | Не подходит как отдельный учебный ассистент без экосистемы Notion      |
| Consensus                   | Есть бесплатные возможности                       | Средняя/высокая     | Хорошее для научных вопросов         | Ориентирован на peer-reviewed literature [14]                  | Полезен для поиска исследований и источников                 | Не решает вопросы ведения студенческого проекта и документации         |
| Учебный чат-бот на базе LMS | Обычно доступен внутри университета               | Средняя             | Зависит от базы знаний               | Может использовать учебные материалы курса                       | Интеграция с курсом и расписанием                            | Часто ограничен FAQ и не умеет вести сложный диалог                    |

Из сравнения видно, что большинство аналогов либо слишком универсальные, либо ориентированы на личные документы, либо решают задачу поиска в интернете. Новый проект должен отличаться тем, что отвечает именно по методологии проектной деятельности и использует проверенную базу знаний, подготовленную под учебный процесс.

# 6 НЕДОСТАЮЩАЯ АНАЛИТИКА ДЛЯ ПЕРВОЙ ИТЕРАЦИИ
Исходя из плюсов и минусов аналогов, в первую итерацию стоит включить не самые сложные функции, а те, которые сразу доказывают ценность проекта и соответствуют критериям оценки.

## 6.1 Что можно включить исходя из плюсов конкурентов
- Ответы с ссылкой на источник. Пользователь должен видеть, на каком документе или фрагменте основан ответ. Это снижает риск недоверия и помогает преподавателю проверить корректность ответа.

- Простые шаблоны для проектной деятельности. Ассистент должен уметь выдавать структуру ТЗ, план проекта, пример user story, пример backlog item, пример критериев приемки и чек-лист для сдачи итерации.

- Поддержка русского языка и понятных формулировок. Так как проект учебный, ответы должны быть короткими, спокойными и без перегруженных технических объяснений.

- Контекст диалога. На базовом уровне ассистент должен помнить тему текущего разговора, чтобы студент мог задавать уточняющие вопросы.

- Сбор обратной связи. После ответа можно дать пользователю простую оценку: «полезно / не полезно» и поле для комментария. Это поможет улучшать базу знаний и промпты.

- Набор тестовых вопросов. Нужно заранее подготовить не менее 20 вопросов по ТЗ, Scrum, Agile, DevOps, документации, ролям и планированию. По ним можно сравнивать версии ассистента.

- Административное обновление базы знаний. Даже если интерфейс администратора будет простым, нужно заложить возможность добавления новых документов и переиндексации.

- Безопасное поведение при нехватке данных. Если в базе знаний нет ответа, ассистент должен честно сообщить об этом и предложить уточнить вопрос, а не придумывать ответ.

## 6.2 Чего стоит избегать
- Слишком сложный интерфейс. В первой версии не нужно делать много настроек модели, температуры, embeddings и количества фрагментов для обычного студента.

- Ответы без источников. Если ассистент отвечает без ссылок на базу знаний, он становится похож на обычный чат-бот и теряет главное отличие проекта.

- Смешивание непроверенных материалов с проверенной базой. Для учебного проекта важно отделять утвержденные материалы от черновиков и внешних ссылок.

- Долгие очереди и медленные ответы. Если ответ появляется слишком поздно, пользователь будет возвращаться к обычным поисковикам или универсальным LLM.

- Отсутствие логов и тестирования. Без логов запросов, найденных фрагментов и оценок пользователя будет трудно понять, почему ассистент ответил плохо.

- Попытка сразу реализовать слишком широкий функционал. На первой итерации лучше качественно сделать RAG по базе знаний, чем добавлять много недоведенных функций.

- Игнорирование вопросов безопасности. Нужно ограничить загрузку файлов, скрывать ключи API, не хранить лишние персональные данные и проверять промпт-инъекции.

# 7 РЕКОМЕНДАЦИИ ПО ПЕРВОЙ ИТЕРАЦИИ
Для первой итерации рекомендуется реализовать классический RAG без сложной агентной логики. Такой вариант проще объяснить, проще протестировать и проще показать заказчику. Минимальная схема должна включать загрузку документов, разбиение на фрагменты, создание embeddings, сохранение в векторную базу, поиск релевантных фрагментов и генерацию ответа с указанием источников.

Таблица 6 - Рекомендуемый состав первой итерации

| **Компонент**        | **Рекомендуемое решение**                                               | **Причина выбора**                                        | **Результат**                            |
|----------------------|-------------------------------------------------------------------------|-----------------------------------------------------------|------------------------------------------|
| База знаний          | Методички, статьи, материалы по Agile, Scrum, DevOps, ТЗ и документации | Соответствует теме проекта и критериям оценки             | Подготовленный набор документов          |
| Разбиение документов | Фрагменты 500-1000 токенов с небольшим overlap                          | Сохраняет смысл фрагмента и не перегружает запрос         | Удобные chunks для поиска                |
| Векторная база       | Chroma для простого MVP или Qdrant для более серьезного прототипа       | Оба варианта подходят для RAG и имеют понятную интеграцию | Векторизованная база знаний              |
| RAG-фреймворк        | LlamaIndex или LangChain                                                | Ускоряют сборку pipeline и работу с документами           | Рабочая цепочка «вопрос - поиск - ответ» |
| LLM                  | API-модель или локальная open-source модель по возможностям команды     | Позволяет быстро проверить идею без обучения модели       | Генерация ответов по контексту           |
| Интерфейс            | Telegram-бот или простая веб-страница                                   | Быстро развернуть и удобно тестировать студентам          | Доступный прототип ассистента            |
| Оценка качества      | 20+ тестовых вопросов и ручная проверка                                 | Требование ТЗ и основа для улучшений                      | Таблица результатов тестирования         |
| Обратная связь       | Кнопка оценки ответа и комментарий                                      | Помогает находить слабые места                            | Данные для второй итерации               |

Рекомендуемая логика ответа ассистента должна быть простой. Сначала система ищет материалы в базе знаний. Затем она формирует ответ только по найденному контексту. В конце ответа желательно указать источники или названия документов. Если релевантных фрагментов нет, ассистент должен написать, что в базе знаний нет достаточной информации.

Для демонстрации минимального результата достаточно реализовать скрипт, который принимает вопрос, ищет 3-5 релевантных фрагментов и возвращает ответ. Для базового результата этот же pipeline нужно подключить к интерфейсу. Для оптимального результата можно добавить историю диалога, сбор обратной связи и страницу администратора для обновления базы знаний.

# ЗАКЛЮЧЕНИЕ
В ходе аналитики были рассмотрены основные подходы к разработке ИИ-ассистента для студентов: классический RAG, Agentic RAG, GraphRAG, fine-tuning и long-context LLM. Для первой итерации наиболее подходящим является классический RAG, так как он напрямую соответствует ТЗ, позволяет работать с проверенной базой знаний и не требует сложной инфраструктуры.

Также были рассмотрены векторные базы данных, фреймворки для RAG-пайплайна и аналоги на рынке. Сравнение показало, что ценность нового проекта должна быть не в универсальности, а в узкой направленности на проектную деятельность студентов. Поэтому в первую версию рекомендуется включить ответы по источникам, простые шаблоны проектной документации, тестовый набор из 20+ вопросов, сбор обратной связи и понятный интерфейс.

Такой объем работ позволяет получить минимальный и базовый результат, указанный в ТЗ, а также подготовить основу для дальнейшего развития: контекст диалога, административное обновление базы знаний, расширенную оценку качества и более сложные RAG-подходы.
