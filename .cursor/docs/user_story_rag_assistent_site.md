# 1. Назначение документа
В данном документе описаны user story для проекта по разработке ИИ-ассистента для студентов. Проект выполняется в формате веб-сайта. Пользователь должен открыть сайт в браузере, задать вопрос по проектной деятельности и получить ответ, основанный на подготовленной базе знаний.

User story нужны для того, чтобы зафиксировать основные действия пользователей, ожидаемую пользу и критерии приемки. На основе этих историй можно планировать первую итерацию разработки, распределять задачи внутри команды и проверять готовность функционала.

# 2. Участники и роли системы
| **Роль**                    | **Описание**                                                                                                              |
|-----------------------------|---------------------------------------------------------------------------------------------------------------------------|
| Студент                     | Основной пользователь сайта. Задает вопросы по проектной деятельности, документации, Agile, Scrum, DevOps и планированию. |
| Участник проектной команды  | Использует сайт для уточнения задач, ролей, этапов работы и подготовки учебного проекта.                                  |
| Преподаватель или наставник | Может использовать сайт как справочный инструмент и проверять, насколько ответы ассистента соответствуют базе знаний.     |
| Администратор базы знаний   | Загружает документы, обновляет материалы и следит за актуальностью базы знаний.                                           |
| Разработчик                 | Настраивает RAG-пайплайн, интерфейс сайта, тесты качества ответов и сбор обратной связи.                                  |

# 3. Основная пользовательская ценность
Главная ценность сайта заключается в том, что студент получает быстрые и понятные ответы не просто от обычной языковой модели, а от ассистента, который использует проверенную базу знаний. Это снижает риск неточных ответов и помогает студенту лучше вести учебный проект.

Сайт должен быть простым: пользователь видит поле для вопроса, примеры запросов, область ответа, источники и возможность оценить результат. Для первой версии важно не перегружать интерфейс сложными настройками.

# 4. User story для веб-сайта
## US-01. Открытие главной страницы сайта
Формулировка: как пользователь сайта, я хочу открыть главную страницу в браузере и сразу понять назначение сервиса, чтобы начать работу без установки дополнительных программ.

Приоритет: Must have. Итерация: 1 итерация.

**Критерии приемки:**

- сайт открывается в браузере без дополнительной установки;

- на главной странице есть краткое описание ассистента;

- пользователь видит поле для ввода вопроса;

- на странице есть примеры вопросов, которые можно задать ассистенту;

- интерфейс не перегружен лишними элементами.

## US-02. Отправка вопроса ассистенту
Формулировка: как студент, я хочу ввести вопрос на сайте и отправить его ассистенту, чтобы быстро получить помощь по проектной деятельности.

Приоритет: Must have. Итерация: 1 итерация.

**Критерии приемки:**

- пользователь может ввести вопрос в текстовое поле;

- кнопка отправки запроса доступна и понятна;

- после отправки сайт показывает, что запрос обрабатывается;

- ответ выводится на той же странице;

- при пустом вопросе сайт просит ввести текст запроса.

## US-03. Ответ на основе базы знаний
Формулировка: как студент, я хочу получить ответ, сформированный на основе загруженных материалов, чтобы использовать проверенную информацию, а не случайный ответ.

Приоритет: Must have. Итерация: 1 итерация.

**Критерии приемки:**

- система ищет подходящие фрагменты в векторной базе данных;

- ассистент формирует ответ с учетом найденных материалов;

- ответ написан простым языком;

- ответ соответствует вопросу пользователя;

- если подходящие материалы найдены, они используются при генерации ответа.

## US-04. Помощь в составлении технического задания
Формулировка: как студент, я хочу спросить у ассистента, как составить техническое задание, чтобы понять, какие разделы должны быть в документе.

Приоритет: Must have. Итерация: 1 итерация.

**Критерии приемки:**

- ассистент объясняет назначение технического задания;

- ассистент перечисляет основные разделы технического задания;

- ассистент может привести пример простой формулировки;

- ответ подходит для учебного проекта;

- ответ не содержит лишних сложных терминов без объяснения.

## US-05. Помощь в планировании проекта
Формулировка: как участник проектной команды, я хочу получить рекомендации по планированию задач, чтобы правильно распределить работу и сроки в команде.

Приоритет: Must have. Итерация: 1 итерация.

**Критерии приемки:**

- ассистент предлагает разбиение проекта на этапы;

- ассистент объясняет, какие задачи можно выполнить сначала;

- ответ может включать примерный план работ;

- ассистент учитывает учебный формат проекта;

- ответ помогает понять, что делать команде дальше.

## US-06. Объяснение Agile, Scrum и DevOps
Формулировка: как студент, я хочу задать вопрос по Agile, Scrum или DevOps, чтобы понять основные подходы к проектной деятельности.

Приоритет: Should have. Итерация: 1 итерация.

**Критерии приемки:**

- ассистент дает краткое объяснение выбранного подхода;

- ассистент перечисляет основные роли, этапы или практики;

- ответ связан с учебным проектом;

- пользователь может задать уточняющий вопрос;

- ассистент не уходит в неподходящие технические детали.

## US-07. Отображение источников ответа
Формулировка: как студент, я хочу видеть источники, на которых основан ответ, чтобы проверять информацию и больше доверять ассистенту.

Приоритет: Must have. Итерация: 1 итерация.

**Критерии приемки:**

- под ответом отображаются найденные источники;

- для источника указано название документа или раздела;

- источники относятся к заданному вопросу;

- пользователь понимает, откуда взята информация;

- если источников нет, сайт сообщает об этом отдельно.

## US-08. Честное сообщение при отсутствии информации
Формулировка: как студент, я хочу получить сообщение, если в базе знаний нет нужных данных, чтобы не использовать выдуманную информацию.

Приоритет: Must have. Итерация: 1 итерация.

**Критерии приемки:**

- если релевантные материалы не найдены, ассистент сообщает об этом;

- ассистент не выдает неподтвержденный ответ как факт;

- сайт предлагает переформулировать вопрос;

- пользователь может отправить новый запрос;

- сообщение написано понятно и без технических подробностей.

## US-09. История диалога в рамках сессии
Формулировка: как студент, я хочу видеть предыдущие вопросы и ответы на странице сайта, чтобы продолжать диалог без потери контекста.

Приоритет: Should have. Итерация: 2 итерация.

**Критерии приемки:**

- на странице отображается история переписки;

- видны сообщения пользователя и ответы ассистента;

- новые сообщения добавляются в конец диалога;

- при уточняющем вопросе ассистент может учитывать контекст текущей сессии;

- после обновления страницы поведение истории описано и понятно пользователю.

## US-10. Оценка качества ответа
Формулировка: как студент, я хочу оценить ответ ассистента на сайте, чтобы помочь команде улучшать качество системы.

Приоритет: Should have. Итерация: 2 итерация.

**Критерии приемки:**

- под ответом есть кнопки оценки, например «полезно» и «не полезно»;

- пользователь может оставить короткий комментарий;

- оценка сохраняется в системе;

- разработчик может посмотреть собранные оценки;

- обратная связь используется для улучшения базы знаний и поиска.

## US-11. Адаптация сайта под разные устройства
Формулировка: как пользователь сайта, я хочу использовать ассистента с ноутбука, планшета или телефона, чтобы получать помощь в удобном формате.

Приоритет: Should have. Итерация: 2 итерация.

**Критерии приемки:**

- основные элементы сайта корректно отображаются на разных размерах экрана;

- поле ввода и кнопка отправки доступны на мобильном устройстве;

- ответ ассистента удобно читать с телефона;

- таблицы или длинные блоки текста не ломают страницу;

- пользователь может отправить новый вопрос без горизонтальной прокрутки.

## US-12. Загрузка документов в базу знаний
Формулировка: как администратор базы знаний, я хочу загрузить новый документ через интерфейс или служебный механизм, чтобы обновлять знания ассистента без изменения основной логики сайта.

Приоритет: Should have. Итерация: 2 итерация.

**Критерии приемки:**

- администратор может добавить документ в базу знаний;

- система обрабатывает документ и разбивает его на фрагменты;

- фрагменты сохраняются в векторной базе данных;

- новый документ используется при последующих запросах;

- при ошибке загрузки система сообщает причину.

## US-13. Управление актуальностью базы знаний
Формулировка: как администратор базы знаний, я хочу обновлять или удалять устаревшие материалы, чтобы поддерживать качество и актуальность ответов.

Приоритет: Could have. Итерация: 3 итерация.

**Критерии приемки:**

- администратор видит список загруженных документов;

- устаревший документ можно заменить новой версией;

- документ можно удалить из базы знаний;

- после изменения выполняется повторная индексация;

- ассистент не должен использовать удаленные материалы.

## US-14. Проверка ассистента на тестовых вопросах
Формулировка: как разработчик, я хочу запустить проверку ассистента на наборе тестовых вопросов, чтобы оценить качество ответов перед демонстрацией проекта.

Приоритет: Must have. Итерация: 1 итерация.

**Критерии приемки:**

- подготовлен набор из 20 или более тестовых вопросов;

- для каждого вопроса фиксируется ответ ассистента;

- ответы проверяются на релевантность;

- результаты проверки сохраняются в таблице или отчете;

- по итогам проверки команда понимает, что нужно улучшить.

# 6. Итоговая формулировка
Основная user story проекта может быть сформулирована следующим образом:

**Как студент, работающий над учебным проектом, я хочу пользоваться сайтом с ИИ-ассистентом, который отвечает на вопросы по проектной деятельности на основе проверенной базы знаний, чтобы быстрее разбираться в документации, планировании задач, методологиях и командной работе.**

В первой итерации основное внимание следует уделить не сложному дизайну, а рабочему сценарию: пользователь открывает сайт, задает вопрос, получает ответ, видит источники и может оценить качество результата. Такой подход позволит быстро показать пользу проекта и затем улучшать систему на следующих этапах.
